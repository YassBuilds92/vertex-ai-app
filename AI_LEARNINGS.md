# AI LEARNINGS (ai-studio)

- [Generated apps / le bundling runtime dans la function Vercel ne doit plus porter le preview produit] -> [Cause] En production, la function backend empaquetee ne retrouve pas toujours `src/generated-app-sdk.tsx` ni `react/jsx-runtime` au moment de builder une draft generated app. Le meme bundling peut donc marcher localement et casser uniquement dans la function Vercel, ce qui rend le preview bundle instable comme chemin canonique. -> [Solution/Code a utiliser] Basculer `GeneratedAppHost` sur un rendu natif via `GeneratedAppCanvas`, garder le bundle comme diagnostic best-effort avec `bundleStatus`, et rendre `publishGeneratedApp()` independant du `bundleCode` tant que `sourceCode|sourceUrl` reste exploitable. Verrouiller ensuite le contrat avec `npx tsx test-generated-app-lifecycle.ts` et `npx tsx test-generated-app-stream.ts`.
- [Generated apps / ne pas resoudre le bundle depuis `process.cwd()` ni laisser un faux `failed` pour un skip d'environnement] -> [Cause] Le bundling de draft dependait de `process.cwd()` pour `esbuild.stdin.resolveDir`, ce qui est fragile en environnement empaquete. Quand la function ne retrouve pas `./src/generated-app-sdk.tsx` ou `react/jsx-runtime`, le produit montrait un faux echec bundle alors que seule la previsualisation optionnelle etait indisponible. -> [Solution/Code a utiliser] Dans `server/lib/generated-apps.ts`, fixer `absWorkingDir`/`resolveDir` depuis `import.meta.url`, garder une importation reelle de `../../src/generated-app-sdk.tsx` pour forcer le packaging du SDK partage, et reclasser ces erreurs de resolution connues via `shared/generated-app-bundle.ts` en `bundleStatus='skipped'` sans `buildLog` surface. Verrouiller avec `npx tsx test-generated-app-bundle-state.ts` et `npx tsx test-generated-app-stream.ts`.
- [Generated app / `uiSchema` non-select ne doit jamais passer par `options.length`] -> [Cause] Dans `server/lib/generated-apps.ts`, `sanitizeFields()` calculait `options` uniquement pour les champs `select`, puis lisait quand meme `options.length` pour tous les types de champ. Une app generee avec un `textarea` ou `text` faisait donc echouer `/api/generated-apps/create` avec `Cannot read properties of undefined (reading 'length')`. -> [Solution/Code a utiliser] Garder `options` comme tableau vide hors `select` ou tester `Array.isArray(options)` avant `.length`, puis verrouiller la regression avec `npx tsx test-generated-app-manifest.ts`.
- [Reparation des sessions derivees / ne pas lancer la reconstruction trop tot] -> [Cause] La reparation `collectionGroup(messages)` peut partir avant que les catalogues `agents` et `generatedApps` aient fini leur chargement distant ou leur fallback local. Un `sessionId` de type `agent-*` ou `gapp-*` peut alors etre reconstruit sans workspace associe et retomber en session standard. -> [Solution/Code a utiliser] Dans `src/App.tsx`, garder des flags `hasLoadedRemoteAgents` et `hasLoadedRemoteGeneratedApps`, puis ne lancer `repairMissingSessionShells()` qu'une fois `sessions + agents + generatedApps` resolves (snapshot ou fallback). 
- [Auth Firebase / ne pas laisser le shell bloque sur `Chargement du studio...`] -> [Cause] Si `onAuthStateChanged()` tarde ou ne repond pas dans certains environnements headless, `src/App.tsx` garde `isAuthReady=false` et l'app reste indefiniment sur le spinner de demarrage, sans jamais montrer l'empty state. -> [Solution/Code a utiliser] Ajouter un fallback `setTimeout` court autour du listener auth dans `src/App.tsx`; si le callback n'a pas resolu apres quelques secondes, logguer un warning et basculer `setIsAuthReady(true)` pour afficher au minimum le shell/empty state, puis laisser le callback reel reprendre la main s'il arrive plus tard.
- [Cowork Apps / un lobby `100dvh` peut encore couper le bas sans scroll] -> [Cause] Mettre `overflow-hidden` sur une vue plein ecran ne suffit pas si on empile hero + spotlight + dock + creation en une seule colonne verticale. Sur un viewport desktop standard, le footer se fait couper meme si la page ne scrolle plus. -> [Solution/Code a utiliser] Dans `src/components/AgentsHub.tsx`, garder une grille `auto / minmax(0,1fr) / auto`, compacter via `window.innerHeight`, basculer le footer desktop en `lg:grid-cols-[minmax(0,1fr)_minmax(300px,0.42fr)]`, et remonter le CTA d'ouverture dans le header du dock pour liberer la hauteur centrale.
- [Lobby d'apps mobile / un wrap libre coupe facilement les icones et la headline] -> [Cause] Sur la nouvelle vue plein ecran `Cowork Apps`, une headline trop large (`max-width`/font-size trop generaux) et une liste d'apps en `flex-wrap` avec largeurs fixes provoquaient encore du clipping sur mobile 430 px. -> [Solution/Code a utiliser] Pour une vue lobby minimaliste, borner la headline avec un `max-w` tres court en `ch`, utiliser une taille `clamp(...)` plus basse sur mobile, puis rendre la liste d'apps en vraie grille mobile (`grid-cols-3`) au lieu d'un wrap libre.
- [QA frontend locale / fallback fiable si Playwright MCP est bloque] -> [Cause] Sur cette machine, la validation navigateur automatisee via Playwright MCP peut echouer avant meme l'ouverture du navigateur avec `EPERM: operation not permitted, mkdir 'C:\\Windows\\System32\\.playwright-mcp'`. En plus, `msedge.exe` n'est pas present dans le `PATH`, donc les commandes shell directes echouent si on n'utilise pas son chemin absolu. -> [Solution/Code a utiliser] Pour les validations visuelles locales, monter un harness HTML/TSX dans `tmp/`, servir le repo via Vite, puis capturer le rendu avec `C:\\Program Files (x86)\\Microsoft\\Edge\\Application\\msedge.exe --headless=new --window-size=... --screenshot=... http://127.0.0.1:4173/tmp/...`. Ouvrir ensuite les PNG avec l'outil image pour confirmer le rendu desktop/mobile.
- [Cowork / la persistance UI seule ne suffit pas pour les follow-ups de correction] -> [Cause] Un message Cowork pouvait bien etre persiste avec `activity`, `runMeta` et `attachments`, mais `buildApiHistoryFromMessages()` ne renvoyait ensuite au modele que `content` + `attachments`. Sur un follow-up comme "le lien est mauvais", Cowork perdait donc la trace textuelle explicite du `release_file`, des URLs deja publiees et du contexte de livraison, puis repartait trop facilement en regeneration. -> [Solution/Code a utiliser] Ajouter dans `src/utils/chat-parts.ts` un mode `includeCoworkMemory` qui injecte pour les messages modele Cowork une memoire textuelle compacte (`runState`, `phase`, `artifactState`, derniers `tool_result`/warnings, URLs des livrables/pieces jointes). Activer ce mode uniquement sur le chemin `/api/cowork` depuis `src/App.tsx`, afin de garder le chat standard inchange.

- [Cowork / un fichier publie doit devenir un evenement SSE typé, pas seulement un lien texte] -> [Cause] Tant que `api/index.ts` emettait seulement `text_delta` et `tool_result`, un `release_file` reussi ne devenait pas une vraie piece jointe UI. Le frontend Cowork recevait au mieux une URL dans le texte final, sans `mimeType` ni type de media pour nourrir `AttachmentGallery`. -> [Solution/Code a utiliser] Faire renvoyer a `release_file` un payload riche (`url`, `path`, `fileName`, `mimeType`, `attachmentType`, `fileSizeBytes`), emettre ensuite un evenement SSE dedie `released_file`, puis hydrater cet evenement dans `src/utils/cowork.ts` vers `msg.attachments` persistantes.
- [Release file / la preview media depend d'un mime fiable, pas d'un `application/octet-stream`] -> [Cause] `server/lib/path-utils.ts` ne couvrait pas plusieurs extensions de livraison reelles (`.mp4`, `.webm`, `.mov`, `.m4a`, etc.). Un fichier publie pouvait alors perdre son type utile lors de l'upload et compliquer l'inference audio/video cote UI. -> [Solution/Code a utiliser] Etendre `getMimeType()` avec les formats media vraiment livres par Cowork, puis completer l'inference partagee via `shared/released-artifacts.ts`.
- [Cowork / generate_music_audio ne doit jamais etre scope global] -> [Cause] Dans `api/index.ts`, `getToolFailureScope()` ne specialisait pas `generate_music_audio`, donc deux echecs sans rapport ou deux reformulations d'un meme essai Lyria 3 incriminaient `generate_music_audio:global`. L'anti-boucle finissait par bloquer toute generation musicale du run apres seulement deux echecs proches, meme si le prompt changeait reellement. -> [Solution/Code a utiliser] Utiliser un scope par `model + prompt family` via `getCoworkToolFailureScope()` pour `generate_music_audio` (et les autres media tools proches), avec `exactKey` base sur le prompt normalise et `familyKey` base sur des tokens d'ancrage du brief.
- [Lyria 3 / un `Internal server error` n'est pas un echec definitif de prompt] -> [Cause] `retryWithBackoff()` ne classait pas `500 / Internal server error` comme incident serveur retryable, et la boucle Cowork ne considerait comme transitoires que `web_search` / `web_fetch`. Resultat: un hoquet serveur Lyria 3 comptait comme un "vrai" echec outil, puis finissait par declencher l'anti-boucle. -> [Solution/Code a utiliser] Dans `server/lib/google-genai.ts`, traiter `500`, `502`, `internal server error`, `timeout` et assimilés comme `server` retryables. Dans `api/index.ts`, utiliser `isTransientCoworkToolIssue()` pour marquer aussi `generate_music_audio`, `generate_tts_audio`, `generate_image_asset` et `create_podcast_episode` comme transitoires sur ces erreurs.
- [Lyria 3 / policy block = reformulation guidee, pas crash brut] -> [Cause] Quand Vertex renvoyait `The prompt contains sensitive words that violate Google's Generative AI Prohibited Use policy`, le tool `generate_music_audio` throwait une erreur brute. Cowork recevait juste un echec opaque sans guidance utile pour reformuler. -> [Solution/Code a utiliser] Dans `server/lib/media-generation.ts`, detecter `isLyriaPolicyBlockedError()`. Puis dans le tool `generate_music_audio` de `api/index.ts`, retourner `success:false`, `recoverable:true`, avec une consigne explicite: repartir sur un brief neutre (`style`, `humeur`, `tempo`, `instruments`, `structure`, `langue`) et eviter imitation d'artiste / formulations sensibles.
- [Sidebar gauche / gros CTA dans la colonne = espace perdu] -> [Cause] Meme avec un seul bouton de creation, le laisser comme une ligne complete dans `src/components/SidebarLeft.tsx` continuait a manger une bande verticale precieuse. Sur une sidebar etroite, l'historique restait visuellement tasse. -> [Solution/Code a utiliser] Deplacer l'action `onNewChat` hors de la colonne historique, idealement dans la barre haute de `src/App.tsx` en bouton carre discret pres du menu. Puis retirer entierement la ligne CTA de la sidebar et compacter encore header/modes/footer pour rendre la hauteur a l'historique.
- [Sidebar gauche / historique trop ecrase par les modes] -> [Cause] Dans `src/components/SidebarLeft.tsx`, la zone haute cumulait un header assez grand, des cartes de mode hautes avec labels qui wrapaient, puis un bouton `+` duplique dans chaque mode. Sur petit viewport, cela volait inutilement la hauteur reservee a l'historique. -> [Solution/Code a utiliser] Compacter la pile haute: reduire la largeur-wrap des modes via une sidebar un peu plus large, diminuer padding/gap/hauteur des rows de mode, supprimer les `+` par-mode, puis creer un seul bouton contextuel `onNewChat` lie a `activeMode` (`modeActionCopy` + `activeModeMeta`) avant la section historique.
- [Chat / fin de stream qui fait sursauter le message final] -> [Cause] En chat standard, le bloc `streaming` et le vrai message final n'etaient pas la meme instance UI. A la fin du flux, `isLoading` retirait d'abord le bloc temporaire, puis le message final attendait parfois Firestore avant d'apparaitre et rejouait en plus l'animation d'entree de `MessageItem`, d'ou un effet de "sursaut". -> [Solution/Code a utiliser] Inserer immediatement le `modelMessage` final dans `optimisticMessages` avant d'attendre Firestore, puis passer un flag `disableEntranceAnimation` a `MessageItem` pour ce message precis afin de neutraliser son `initial` Motion pendant la transition de fin de stream.
- [Chat / thinking doit se replier automatiquement a la livraison] -> [Cause] `src/App.tsx` marquait le message final comme ouvert avec `setExpandedThoughts(...[modelMsgId]: true)` des que des thoughts existaient, donc le volet reasoning restait deploye apres la reponse. -> [Solution/Code a utiliser] A la fin du stream, fermer `streamingThoughtsExpanded`, puis initialiser l'etat du message final avec `[modelMsgId]: false` pour garder le bouton et le contenu disponibles, mais replies par defaut.
- [Chat / auto-scroll qui arrache la lecture pendant la reponse] -> [Cause] Un `useEffect` de `src/App.tsx` appelait `messagesEndRef.current?.scrollIntoView(...)` a chaque chunk (`streamingContent`, `streamingThoughts`, `isLoading`), donc toute tentative de remonter lire l'historique etait annulee des qu'une reponse arrivait. -> [Solution/Code a utiliser] Remplacer l'auto-scroll global par un mode sticky conditionnel: suivre le bas seulement si le conteneur principal est deja proche du bas (`isScrolledNearBottom()` avec un seuil d'environ `96px`), memoriser cet etat dans `shouldAutoScrollRef`, le reinitialiser lors d'un changement de session, et ne re-scroller pendant le stream que si ce flag est encore vrai.
- [Fenetre visible du chat / ne pas casser modifier-renvoyer] -> [Cause] Si on limite l'affichage avec un simple `slice(-15)` puis qu'on continue a passer l'index visible a `MessageItem`, les handlers `handleEdit` et `handleRetry` pointent vers les mauvais messages dans `currentMessages`. -> [Solution/Code a utiliser] Garder l'historique complet dans `displayedMessages`, calculer `visibleMessages` et `visibleMessageOffset` dans `src/App.tsx`, rendre seulement `visibleMessages`, puis passer `idx={visibleMessageOffset + index}` aux actions de message.
- [Chat/Cowork / piece jointe visible dans l'UI mais invisible pour le modele] -> [Cause] Le frontend envoyait la piece jointe courante dans `attachments`, mais `/api/chat` et `/api/cowork` reconstruisaient `contents` avec `{ text: message }` seulement et ignoraient ce champ. En plus, l'historique transformait les anciennes pieces jointes en `inlineData` vide des que `base64` avait ete retire apres upload. -> [Solution/Code a utiliser] Introduire un contrat partage de parts (`shared/chat-parts.ts`), serialiser l'historique via `buildApiHistoryFromMessages()` cote `src/App.tsx`, puis reconstruire les vraies parts modele via `buildModelContentsFromRequest()` dans `server/lib/chat-parts.ts`. Pour les images/PDF, utiliser `base64` quand disponible puis fallback sur fetch de l'URL signee; pour YouTube, injecter un contexte texte `Titre + URL` au lieu d'un faux `fileData` video.
- [Empty state qui scroll sans messages] -> [Cause] Dans un layout flex vertical, un `main` scrollable sans `min-h-0` + un spacer de fin (`messagesEndRef`) rendu meme a vide suffit a provoquer un debordement fantome. -> [Solution/Code a utiliser] Dans `src/App.tsx`, mettre le panneau central en `min-h-0`, passer `overflow-y-auto` seulement quand il y a vraiment du contenu, et rendre le spacer `messagesEndRef` uniquement s'il y a des messages, un stream ou un statut de generation.
- [Hero qui ne suit pas les volets lateraux] -> [Cause] Une typo cale sur les breakpoints viewport reagit mal quand la largeur utile change a l'interieur d'un shell app (volets ouverts/fermes) sans changer drastiquement le viewport. -> [Solution/Code a utiliser] Declarer `container-type: inline-size` sur `.studio-empty-state` dans `src/index.css`, puis piloter la headline et les elements d'ambiance avec des valeurs `cqw`/`clamp(...)` pour que le hero suive la vraie largeur disponible du panneau central.
- [Hero empty state / glow premium sans cout JS] -> [Cause] Un empty state visuellement riche peut vite devenir trop charge si le signal repose sur plusieurs cartes animees. Les animations continues pilotees en JS ou par transforms multiples coutent plus qu'un simple effet d'atmosphere. -> [Solution/Code a utiliser] Pour `src/components/StudioEmptyState.tsx`, basculer vers un poster state unique et confier le relief a des mots d'ambiance en fond via CSS (`studio-empty-state__word*`) avec variables theme-aware, `text-shadow` discret, animation lente sur `opacity` seulement et fallback `prefers-reduced-motion`.
- [Podcast duo / deux intervenants mais meme rendu vocal percu] -> [Cause] Un duo TTS peut sembler "une seule voix" si les 2 speakers arrivent avec la meme voix prebuilt ou avec des notes de jeu trop proches. En freeform multi-speaker, des labels trop libres compliquent aussi le routage/diagnostic. -> [Solution/Code a utiliser] Normaliser les speakers via `normalizeGeminiTtsSpeakers()` avec 2 voix garanties distinctes, injecter des notes de jeu contrastees par defaut, generer des aliases alphanumeriques stables (`ttsAlias`) et rerouter les labels via `remapDialogueSpeakerLabelsForTts()` avant `multiSpeakerVoiceConfig.speakerVoiceConfigs`.
- [Lyria 3 / endpoint interactions 404] -> [Cause] Lyria 3 ne passe pas par `https://global-aiplatform.googleapis.com/.../interactions`. La doc officielle Vertex AI utilise `https://aiplatform.googleapis.com/v1beta1/projects/PROJECT_ID/locations/global/interactions`. -> [Solution/Code a utiliser] Dans `server/lib/media-generation.ts`, appeler l'endpoint `aiplatform.googleapis.com` pour `generateLyria3Binary()`. Le smoke test local a ensuite reussi avec `lyria-3-clip-preview` et une sortie `audio/mpeg`.
- [Historique / nouvelle discussion qui apparait puis disparait] -> [Cause] la sidebar des sessions etait remplacee integralement par `onSnapshot(users/{uid}/sessions)` dans `src/App.tsx`. Quand l'ecriture du shell `sessions/{sessionId}` echouait ou n'etait pas encore visible, la session creee localement etait ecrasee par la liste distante, donc l'historique clignotait puis disparaissait. -> [Solution/Code a utiliser] Introduire un cache local de shells dans `src/utils/sessionShells.ts`, sauvegarder chaque session locale avec `pendingRemote: true`, puis utiliser `mergeSessionsWithLocal(user.uid, remoteSessions)` dans le listener Firestore au lieu d'un simple `setSessions(snapshot.docs.map(...))`. Nettoyer aussi le cache local sur suppression via `removeLocalSessionShell()`.

- [Firestore / sessions invisibles entre appareils] -> [Cause] Les documents `users/{uid}/sessions/{sessionId}` pouvaient etre refuses par `firestore.rules` parce que le frontend ecrit maintenant `sessionKind` et parfois `agentWorkspace`, tandis que les messages continuaient a s'ecrire dans `users/{uid}/sessions/{sessionId}/messages`. Resultat : messages presents dans Firestore mais session shell absent, donc aucune conversation visible sur les autres appareils. -> [Solution/Code a utiliser] Dans `firestore.rules`, autoriser `sessionKind` et `agentWorkspace` dans `isValidSession()`. Dans `src/App.tsx`, lancer une reparation one-shot via `collectionGroup(db, 'messages')` + `where('userId', '==', uid)` pour reconstruire les shells manquants avec `buildRecoveredSessionShell()` depuis `src/utils/sessionRecovery.ts`.
- [Firestore / Cowork rich payload degrade] -> [Cause] Les regles validaient un ancien schema `runMeta` (`webSearches`, `webFetches`, etc.) alors que le runtime envoie `searchCount`, `fetchCount`, `sourcesOpened`, `domainsOpened`, `artifactState`, `stalledTurns`, `mode`. -> [Solution/Code a utiliser] Aligner `isValidCoworkRunMeta()` dans `firestore.rules` sur le vrai contrat `CoworkRunMeta` de `api/index.ts` / `src/utils/cowork.ts` pour eviter le fallback silencieux en mode `legacy`.

Solution : Brancher `handleSend` dans `App.tsx` pour appeler `/api/generate-image` ou `/api/generate-video` spécifiquement selon le mode actif, au lieu de tout envoyer à `/api/chat`.

- [Modèle rapide/éco] -> `gemini-3.1-flash-lite-preview` (Sortie Mars 2026) à privilégier pour les tâches de fond comme le raffinement de prompt.

## SDK Google GenAI (Migration @google-cloud/vertexai -> @google/genai)
- [Deprecation VertexAI Class] -> Le SDK `@google-cloud/vertexai` (classe `VertexAI`) est deprecated depuis juin 2025, suppression prévue juin 2026. Le runtime lève un warning/erreur "The VertexAI class and all..." qui crashe l'API en 500. **Solution** : Migrer vers `@google/genai` avec `vertexai: true`. Installation : `npm install @google/genai`.
- [Initialisation @google/genai] -> `new GoogleGenAI({ vertexai: true, project: 'ID', location: 'global', googleAuthOptions: { credentials: gcpCredentials } })`. NE PAS utiliser `new VertexAI(...)`.
- [API generateContent] -> Ancienne syntaxe `client.getGenerativeModel({model}).generateContent({contents, systemInstruction, generationConfig})`. **Nouvelle** : `ai.models.generateContent({ model, contents, config: { systemInstruction, temperature, topP, topK, maxOutputTokens, tools } })`. La réponse : `result.text` (au lieu de `result.response.candidates[0]...`).
- [API generateContentStream] -> `ai.models.generateContentStream({ model, contents, config })`. Itération : `for await (const chunk of response) { chunk.text }`. Pour les thoughts : accéder via `chunk.candidates[0].content.parts` et vérifier `part.thought`.
- [Tools nouveau SDK] -> Utiliser `{ googleSearch: {} }`, `{ codeExecution: {} }` dans `config.tools` (et non `google_search_retrieval`).
- [Error Handling SSE] -> Ne pas setter les headers SSE (`Content-Type: text/event-stream`) AVANT d'avoir vérifié que le modèle/config est valide. Utiliser un flag `headersSent` pour savoir si on doit renvoyer du JSON (erreur avant stream) ou un event SSE d'erreur (erreur pendant stream).

## Déploiement Vercel (Express + Vite)
- [Écran Blanc / Routage] -> Sur Vercel, ne pas utiliser le champ legacy `builds` dans `vercel.json`. Préférer les `rewrites` pour tout rediriger vers `server.ts`.
- [Service Files Statiques] -> Dans `server.ts`, s'assurer que `app.use(express.static(distPath))` et le fallback `app.get('*', ...)` sont enregistrés en dehors de toute fonction `listen()` manuelle, car Vercel ignore les appels à `listen`.
- [Mur de Mot de Passe] -> Pour protéger tout le site (frontend + API), router toutes les requêtes `/(.*)` vers le serveur Express via `vercel.json`. S'assurer que `SITE_PASSWORD` est défini dans les variables d'environnement Vercel.

## UI & Thèmes

- [Mode OLED] -> Le noir absolu est `#000000`. S'assurer de l'appliquer via `html.oled` avec `!important` pour écraser les variables par défaut.

## Firebase & Firestore
- [Champs Undefined] -> Firestore `addDoc`/`setDoc` rejette les valeurs `undefined`. Utiliser un utilitaire `cleanForFirestore` pour supprimer récursivement les propriétés `undefined`.
- [Validation Zod] -> Attention aux types `z.string().optional()` : ils n'acceptent pas `null`. Préférer `nullable().optional()` si le frontend peut envoyer des valeurs nulles, ou s'assurer d'envoyer `undefined`.
- [Incohérence API] -> Toujours vérifier que les noms de champs envoyés par `fetch` (`body: JSON.stringify({ prompt: ... })`) correspondent exactement aux schémas Zod du backend (`prompt` vs `message`).
- [Erreur Permissions] -> Si `Missing or insufficient permissions` persiste après avoir desserré les règles, vérifier si les règles locales `firestore.rules` ont bien été déployées sur le cloud via `firebase deploy --only firestore:rules`.
- [Connectivité Firestore (Net::ERR_...)] -> Les erreurs `ERR_BLOCKED_BY_CLIENT` ou `ERR_QUIC_PROTOCOL_ERROR` sont souvent dues à des AdBlockers ou des blocages réseau UDP. **Solution** : Forcer le mode HTTP dans `firebase.ts` avec `experimentalForceLongPolling: true` ET `experimentalAutoDetectLongPolling: false`.
- [Conflit Project ID / Database ID] -> Une erreur `403 Forbidden` ou `Blocked by client` peut survenir si le code frontend tente d'accéder à une base de données d'un projet différent (ex: Authentification sur Projet A mais Firestore sur Projet B). Toujours vérifier que `projectId` et `firestoreDatabaseId` dans `firebase-applet-config.json` pointent vers le même projet GCP.
- [Clés React Doublons] -> Dans les listes virtualisées, s'assurer d'utiliser `msg.id` comme clé plutôt que l'index pour éviter les problèmes de rendu (écran blanc).
- [Erreur de Référence Initialisation] -> Dans `App.tsx`, s'assurer que `displayedMessages` (utilisé par `rowVirtualizer`) est déclaré AVANT l'initialisation du virtualiseur pour éviter une `ReferenceError` (Temporal Dead Zone).
- [Bug Visuel Renvoyer/Regénérer] -> Si un message vide apparaît lors d'une régénération, vérifier `handleSend`. Il ne faut pas créer de nouveau message optimiste ou Firestore si `overrideMessages` est présent. La `message` envoyée à l'API doit être le dernier élément de l'historique, et l'historique doit être tronqué d'un élément pour éviter les doublons côté serveur.
- [Firestore Error] -> `addDoc` rejette les propriétés `undefined`. Toujours utiliser `value || null` pour les champs optionnels (ex: `iconUrl`).
